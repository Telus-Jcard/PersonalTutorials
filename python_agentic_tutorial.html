Installation & Setup

Before writing any agent code, let's set up a clean local environment. We'll use a virtual environment to isolate dependencies — a good habit that also reflects the Single Responsibility Principle.

Key Directory Components

__init__.py Files — Package Initialization

Python uses __init__.py files to mark directories as packages, enabling imports from those directories. Benefits include:

  • Package Recognition: Allows Python to treat directories as importable modules (e.g., from agents import base).
  • Namespace Control: Use __init__.py to define what gets exported via __all__, controlling your public API.
  • Centralized Initialization: Run setup code when a package is imported (e.g., logging, config defaults).
  • Relative Imports: Enable clean relative imports like from .base import BaseAgent within the package.

__pycache__ Folder — Compiled Bytecode Cache

Python automatically creates a __pycache__ folder in each directory containing compiled bytecode (.pyc files). Benefits include:

  • Faster Imports: Python caches compiled bytecode, so subsequent imports skip the compilation step and load much faster.
  • Automatic Management: Python recreates .pyc files when source code changes, so you don't manage it manually.
  • Development Convenience: Safe to ignore (add __pycache__/ to .gitignore) — it's not meant for version control, only for runtime optimization.
  • Multi-Python Compatibility: __pycache__ stores bytecode for different Python versions separately (e.g., base.cpython-39.pyc), preventing conflicts.

// 02 — Python Basics

Python Fundamentals for Agents

Agentic systems rely heavily on a few Python patterns. This section covers the essential building blocks — type hints, async/await, dataclasses, and abstract classes — that you'll use throughout.

Type Hints & Dataclasses

Type hints make agent code readable and catch bugs early. dataclasses are perfect for structured agent state.

agent_state.py python 
from dataclasses import dataclass, field
from typing import Optional, List, Dict, Any
from enum import Enum

class AgentStatus(Enum):
    IDLE = "idle"
    THINKING = "thinking"
    ACTING = "acting"
    DONE = "done"
    ERROR = "error"

@dataclass
class Message:
    role: str                          # "user", "assistant", "system"
    content: str
    metadata: Dict[str, Any] = field(default_factory=dict)

@dataclass
class AgentState:
    """Immutable-ish state snapshot for an agent run."""
    messages: List[Message] = field(default_factory=list)
    status: AgentStatus = AgentStatus.IDLE
    context: Dict[str, Any] = field(default_factory=dict)
    error: Optional[str] = None

    def add_message(self, role: str, content: str) -> "AgentState":
        """Returns new state — avoids mutation."""
        from copy import deepcopy
        new_state = deepcopy(self)
        new_state.messages.append(Message(role=role, content=content))
        return new_state

# Usage
state = AgentState()
state = state.add_message("user", "Research quantum computing")
print(state.messages[0].content)  # Research quantum computing

Async / Await

Most LLM API calls are I/O-bound. Using async/await lets your agent run multiple tools concurrently instead of waiting sequentially.

async_tools.py python 
import asyncio
from typing import List

async def fetch_web_result(query: str) -> str:
    """Simulates an async web search call."""
    await asyncio.sleep(0.5)  # replace with real API call
    return f"Results for: {query}"

async def run_tools_concurrently(queries: List[str]) -> List[str]:
    """Run multiple tool calls at the same time — much faster!
    
    Sequential: 3 calls × 0.5s = 1.5s total
    Concurrent: max(0.5s, 0.5s, 0.5s) = 0.5s total
    """
    tasks = [fetch_web_result(q) for q in queries]
    results = await asyncio.gather(*tasks)
    return list(results)

# Run it
async def main():
    results = await run_tools_concurrently([
        "quantum computing",
        "neural networks",
        "reinforcement learning"
    ])
    for r in results:
        print(r)

asyncio.run(main())

// 03 — OOP for Agents

Object-Oriented Patterns

Agentic systems shine when built with abstract base classes. They enforce contracts between components — the foundation of SOLID design.

agents/base.py python 
from abc import ABC, abstractmethod
from typing import Any, Dict, Optional

class BaseTool(ABC):
    """Abstract contract every tool must fulfil."""

    @property
    @abstractmethod
    def name(self) -> str:
        """Unique identifier for this tool."""
        ...

    @property
    @abstractmethod
    def description(self) -> str:
        """Human-readable description for the LLM."""
        ...

    @abstractmethod
    async def run(self, input: Dict[str, Any]) -> str:
        """Execute the tool and return a string result."""
        ...


class BaseAgent(ABC):
    """Abstract contract every agent must fulfil."""

    def __init__(self, tools: list[BaseTool] = None):
        self.tools = tools or []
        self._tool_map = {t.name: t for t in self.tools}

    @abstractmethod
    async def run(self, task: str) -> str:
        """Execute a task and return the result."""
        ...

    def get_tool(self, name: str) -> Optional[BaseTool]:
        return self._tool_map.get(name)
💡
Why Abstract Classes?

Abstract classes enforce that all subclasses implement required methods. If you forget to implement run(), Python raises TypeError at instantiation — catching the bug before runtime.

// 04 — SOLID Principles

Single Responsibility Principle

S
Single Responsibility
A class should have only ONE reason to change.

In agentic systems, this means each agent, tool, or class does one thing. A ResearchAgent researches. A WriterAgent writes. A WebSearchTool searches the web. Nothing more.

❌ VIOLATES SRP
class DoEverythingAgent:
    # Searches, writes, emails,
    # logs — too many jobs!
    def search_web(self, q): ...
    def write_report(self): ...
    def send_email(self): ...
    def log_to_db(self): ...
✅ FOLLOWS SRP
class ResearchAgent:
    async def run(self, task): ...

class WriterAgent:
    async def run(self, task): ...

class EmailTool:
    async def run(self, input): ...

Full Example

agents/researcher.py python 
from agents.base import BaseAgent, BaseTool
from openai import AsyncOpenAI

class ResearchAgent(BaseAgent):
    """Does ONE thing: research a topic using available tools.
    
    It does NOT write reports, send emails, or format outputs.
    Those are other agents' responsibilities.
    """

    def __init__(self, tools: list[BaseTool]):
        super().__init__(tools)
        self.client = AsyncOpenAI()
        self.system_prompt = (
            "You are a research specialist. Given a topic, "
            "gather comprehensive, accurate information. "
            "Return ONLY research findings, not analysis."
        )

    async def run(self, task: str) -> str:
        response = await self.client.chat.completions.create(
            model="gpt-4o-mini",
            messages=[
                {"role": "system", "content": self.system_prompt},
                {"role": "user", "content": task}
            ]
        )
        return response.choices[0].message.content

// 04 — SOLID Principles

Open/Closed Principle

O
Open/Closed Principle
Open for extension, closed for modification.

You should be able to add new agent types or LLM providers without changing existing, working code. Use inheritance and composition to extend behavior.

🔌
Real-world application

You start with OpenAI. Later you want to add Anthropic support. OCP means you extend with a new class — you don't touch the existing OpenAIAgent.

agents/llm_agents.py python 
from abc import ABC, abstractmethod
from openai import AsyncOpenAI
from anthropic import AsyncAnthropic

class LLMAgent(ABC):
    """Base: closed for modification."""

    @abstractmethod
    async def complete(self, prompt: str) -> str: ...


class OpenAIAgent(LLMAgent):
    """Extension 1 — never need to touch this after shipping."""

    def __init__(self, model: str = "gpt-4o-mini"):
        self.client = AsyncOpenAI()
        self.model = model

    async def complete(self, prompt: str) -> str:
        response = await self.client.chat.completions.create(
            model=self.model,
            messages=[{"role": "user", "content": prompt}]
        )
        return response.choices[0].message.content


class AnthropicAgent(LLMAgent):
    """Extension 2 — added later without touching OpenAIAgent."""

    def __init__(self, model: str = "claude-sonnet-4-6"):
        self.client = AsyncAnthropic()
        self.model = model

    async def complete(self, prompt: str) -> str:
        message = await self.client.messages.create(
            model=self.model,
            max_tokens=1024,
            messages=[{"role": "user", "content": prompt}]
        )
        return message.content[0].text


class OllamaAgent(LLMAgent):
    """Extension 3 — local model, zero cloud dependency."""

    def __init__(self, model: str = "llama3.2", base_url: str = "http://localhost:11434"):
        from openai import AsyncOpenAI
        self.client = AsyncOpenAI(api_key="ollama", base_url=f"{base_url}/v1")
        self.model = model

    async def complete(self, prompt: str) -> str:
        response = await self.client.chat.completions.create(
            model=self.model,
            messages=[{"role": "user", "content": prompt}]
        )
        return response.choices[0].message.content

# Orchestrator uses the abstract type — doesn't care which LLM
async def run_task(agent: LLMAgent, task: str) -> str:
    return await agent.complete(task)

// 04 — SOLID Principles

Liskov Substitution Principle

L
Liskov Substitution
Subtypes must be substitutable for their base types.

Any code that works with BaseTool must work with any subclass of BaseTool without knowing the concrete type. Subclasses must honor the contract — same input/output shapes, no unexpected exceptions.

tools/search_tools.py python 
from agents.base import BaseTool
from typing import Any, Dict

class WebSearchTool(BaseTool):
    """Real web search — honors BaseTool contract."""

    @property
    def name(self) -> str:
        return "web_search"

    @property
    def description(self) -> str:
        return "Search the internet for current information."

    async def run(self, input: Dict[str, Any]) -> str:
        query = input["query"]
        # Real implementation would call a search API
        return f"[WebSearch] Results for '{query}': ..."


class MockSearchTool(BaseTool):
    """Test double — fully substitutable for WebSearchTool.
    
    LSP in action: any code expecting BaseTool works with this too.
    Use during testing to avoid real API calls.
    """

    @property
    def name(self) -> str:
        return "web_search"  # Same name!

    @property
    def description(self) -> str:
        return "Mock search for testing."

    async def run(self, input: Dict[str, Any]) -> str:
        return "[Mock] Quantum computing is the future."


# Both work identically from the caller's perspective
async def use_tool(tool: BaseTool, query: str) -> str:
    return await tool.run({"query": query})

# In production:
result = await use_tool(WebSearchTool(), "quantum computing")

# In tests — no API key needed:
result = await use_tool(MockSearchTool(), "quantum computing")

// 04 — SOLID Principles

Interface Segregation Principle

I
Interface Segregation
No class should be forced to implement methods it doesn't use.

Don't build one massive BaseAgent with every possible method. Instead, create small, focused interfaces (protocols/ABCs) that agents opt into.

agents/interfaces.py python 
from abc import ABC, abstractmethod
from typing import Protocol

# ❌ FAT INTERFACE — ISP violation
class FatAgent(ABC):
    @abstractmethod
    async def search(self): ...    # not every agent searches
    @abstractmethod
    async def write(self): ...     # not every agent writes
    @abstractmethod
    async def summarize(self): ... # not every agent summarizes
    @abstractmethod
    async def email(self): ...     # not every agent emails


# ✅ SEGREGATED INTERFACES — ISP compliant
class Runnable(Protocol):
    """Core contract: every agent can be run."""
    async def run(self, task: str) -> str: ...

class Searchable(Protocol):
    """Opt-in: agents that can search the web."""
    async def search(self, query: str) -> str: ...

class Summarizable(Protocol):
    """Opt-in: agents that summarize content."""
    async def summarize(self, content: str) -> str: ...

class Memorizable(Protocol):
    """Opt-in: agents that have persistent memory."""
    def remember(self, key: str, value: str) -> None: ...
    def recall(self, key: str) -> str: ...


# Agents pick ONLY the interfaces they need
class ResearchAgent:
    """Implements Runnable + Searchable only."""
    async def run(self, task: str) -> str:
        findings = await self.search(task)
        return findings

    async def search(self, query: str) -> str:
        return f"Searched: {query}"


class MemoryAgent:
    """Implements Runnable + Memorizable only."""
    def __init__(self):
        self._memory: dict = {}

    async def run(self, task: str) -> str:
        return f"Recalled: {self.recall(task)}"

    def remember(self, key: str, value: str) -> None:
        self._memory[key] = value

    def recall(self, key: str) -> str:
        return self._memory.get(key, "[not found]")

// 04 — SOLID Principles

Dependency Inversion Principle

D
Dependency Inversion
Depend on abstractions, not concrete implementations.

High-level agent orchestrators shouldn't import OpenAI or Anthropic directly. They depend on abstract LLMClient interfaces. The concrete provider is injected from outside.

agents/orchestrator.py python 
from abc import ABC, abstractmethod
from typing import List
from dataclasses import dataclass

# ─ Abstraction layer ─
class LLMClient(ABC):
    """High-level policy: what an LLM client can do."""
    @abstractmethod
    async def complete(self, messages: List[dict]) -> str: ...

class MemoryStore(ABC):
    @abstractmethod
    def save(self, key: str, value: str) -> None: ...
    @abstractmethod
    def load(self, key: str) -> str: ...


# ─ High-level orchestrator depends ONLY on abstractions ─
class AgentOrchestrator:
    """Never imports OpenAI or any concrete class directly."""

    def __init__(
        self,
        llm: LLMClient,          # injected
        memory: MemoryStore,     # injected
    ):
        self.llm = llm
        self.memory = memory

    async def run(self, task: str) -> str:
        context = self.memory.load("context")
        result = await self.llm.complete([
            {"role": "system", "content": f"Context: {context}"},
            {"role": "user", "content": task}
        ])
        self.memory.save("last_result", result)
        return result


# ─ Concrete implementations (details) ─
class OpenAIClient(LLMClient):
    async def complete(self, messages: List[dict]) -> str:
        from openai import AsyncOpenAI
        client = AsyncOpenAI()
        resp = await client.chat.completions.create(
            model="gpt-4o-mini", messages=messages
        )
        return resp.choices[0].message.content

class InMemoryStore(MemoryStore):
    def __init__(self): self._store = {}
    def save(self, key, value): self._store[key] = value
    def load(self, key): return self._store.get(key, "")


# ─ Wire it up at the top level (composition root) ─
orchestrator = AgentOrchestrator(
    llm=OpenAIClient(),
    memory=InMemoryStore(),
)

# Later, swap to Anthropic without touching AgentOrchestrator:
# orchestrator = AgentOrchestrator(llm=AnthropicClient(), ...)

LangGraph

LangGraph models agents as a directed graph of nodes (functions) connected by edges (transitions). State flows through the graph, enabling cycles, retries, and complex routing logic.

install 
$ pip install langgraph langchain langchain-openai
workflows/langgraph_agent.py python 
from typing import TypedDict, Annotated, Sequence
from langgraph.graph import StateGraph, END
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage, AIMessage, BaseMessage
import operator

# ─ State definition (TypedDict for LangGraph) ─
class AgentState(TypedDict):
    messages: Annotated[Sequence[BaseMessage], operator.add]
    next_step: str

# ─ Node functions (SRP: each does ONE thing) ─
def research_node(state: AgentState) -> AgentState:
    """Research node: gather information."""
    llm = ChatOpenAI(model="gpt-4o-mini")
    last_msg = state["messages"][-1].content
    response = llm.invoke([
        HumanMessage(content=f"Research this topic concisely: {last_msg}")
    ])
    return {
        "messages": [AIMessage(content=response.content, name="researcher")],
        "next_step": "write"
    }

def writer_node(state: AgentState) -> AgentState:
    """Writer node: produce a final report."""
    llm = ChatOpenAI(model="gpt-4o-mini")
    research = state["messages"][-1].content
    response = llm.invoke([
        HumanMessage(content=f"Write a clear 3-paragraph summary of: {research}")
    ])
    return {
        "messages": [AIMessage(content=response.content, name="writer")],
        "next_step": "end"
    }

def router(state: AgentState) -> str:
    """Route to the next node based on state."""
    return state.get("next_step", "research")

# ─ Build the graph (OCP: extend with new nodes, don't modify existing) ─
workflow = StateGraph(AgentState)
workflow.add_node("research", research_node)
workflow.add_node("write", writer_node)

workflow.set_entry_point("research")
workflow.add_conditional_edges(
    "research",
    router,
    {"write": "write", "end": END}
)
workflow.add_edge("write", END)

app = workflow.compile()

# ─ Run it ─
if __name__ == "__main__":
    result = app.invoke({
        "messages": [HumanMessage(content="quantum computing")],
        "next_step": "research"
    })
    print(result["messages"][-1].content)

CrewAI

CrewAI uses a crew of agents — each with a defined role, goal, and backstory — working together on tasks. It's the most intuitive for multi-agent delegation and naturally encourages SRP.

install 
$ pip install crewai crewai-tools
workflows/crewai_pipeline.py python 
from crewai import Agent, Task, Crew, Process
from crewai_tools import SerperDevTool
import os

# ─ Tools (SRP: each tool does one thing) ─
search_tool = SerperDevTool()  # requires SERPER_API_KEY

# ─ Agents (SRP: each agent has exactly one role) ─
researcher = Agent(
    role="Research Specialist",
    goal="Find accurate, current information on the given topic",
    backstory="""You are an expert researcher with a talent for 
    finding precise, relevant information from reliable sources.""",
    tools=[search_tool],
    verbose=True,
    llm="gpt-4o-mini"
)

writer = Agent(
    role="Technical Writer",
    goal="Transform research findings into clear, engaging content",
    backstory="""You excel at taking complex research and making 
    it accessible, structured, and compelling.""",
    verbose=True,
    llm="gpt-4o-mini"
)

# ─ Tasks (OCP: add new tasks without modifying agents) ─
research_task = Task(
    description="""Research the topic: {topic}
    Find key facts, recent developments, and important concepts.
    Your output should be detailed research notes.""",
    expected_output="Detailed research notes with key findings",
    agent=researcher
)

write_task = Task(
    description="""Using the research provided, write a comprehensive
    3-section report with: Introduction, Key Findings, Conclusion.""",
    expected_output="A polished 3-section report",
    agent=writer,
    context=[research_task]  # receives researcher's output
)

# ─ Crew — orchestrates the pipeline ─
crew = Crew(
    agents=[researcher, writer],
    tasks=[research_task, write_task],
    process=Process.sequential,
    verbose=True
)

if __name__ == "__main__":
    result = crew.kickoff(inputs={"topic": "agentic AI systems"})
    print(result.raw)

PydanticAI

PydanticAI brings Pydantic's type-safety to AI agents. It ensures LLM outputs conform to exact Python types — no hallucinated fields, no missing keys. Perfect for production APIs.

install 
$ pip install pydantic-ai
workflows/pydantic_agent.py python 
from pydantic_ai import Agent
from pydantic_ai.models.openai import OpenAIModel
from pydantic import BaseModel, Field
from typing import List
import asyncio

# ─ Structured output model (type safety) ─
class ResearchReport(BaseModel):
    """Validated structured output — no surprises."""
    title: str = Field(description="Report title")
    summary: str = Field(description="2-3 sentence executive summary")
    key_points: List[str] = Field(
        description="3-5 key findings",
        min_length=3, max_length=5
    )
    confidence: float = Field(
        description="Confidence score 0-1",
        ge=0.0, le=1.0
    )

# ─ Dependency injection (DIP: agent uses abstract context) ─
class ResearchContext(BaseModel):
    topic: str
    depth: str = "comprehensive"

# ─ Create the agent (OCP: swap models by changing OpenAIModel) ─
agent = Agent(
    model=OpenAIModel("gpt-4o-mini"),
    result_type=ResearchReport,   # enforces typed output
    deps_type=ResearchContext,
    system_prompt="""You are an expert researcher. Analyze the topic
    provided in the context and return a structured research report.
    Be concise and accurate. Assign confidence based on your certainty."""
)

@agent.system_prompt
async def add_context(ctx) -> str:
    """Dynamically inject context into the system prompt."""
    return f"Research depth requested: {ctx.deps.depth}"

async def main():
    context = ResearchContext(topic="quantum computing", depth="overview")
    result = await agent.run(
        f"Research topic: {context.topic}",
        deps=context
    )
    report: ResearchReport = result.data

    # Fully typed — IDE autocomplete works here!
    print(f"Title: {report.title}")
    print(f"Confidence: {report.confidence:.0%}")
    print(f"Key points: {len(report.key_points)}")
    for point in report.key_points:
        print(f"  • {point}")

asyncio.run(main())
✅
Why PydanticAI for Production?

If the LLM returns the wrong number of key_points, a non-float confidence, or missing fields, Pydantic raises a ValidationError immediately — before the bad data reaches your database or UI.

// 06 — Capstone

Putting It All Together

Here's a complete SOLID-compliant multi-agent research pipeline that you can run locally. It combines everything: abstract base classes, dependency injection, segregated interfaces, and a LangGraph workflow.

main.py — complete runnable example python 
"""
Complete SOLID-compliant agentic pipeline.
Run: python main.py

Requires: pip install langgraph langchain langchain-openai python-dotenv
          OPENAI_API_KEY in .env
"""
import asyncio
from abc import ABC, abstractmethod
from typing import TypedDict, List, Annotated, Sequence
from dotenv import load_dotenv
import operator

from langgraph.graph import StateGraph, END
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage, AIMessage, BaseMessage

load_dotenv()

# ══════════════════════════════════════════════════════
# ABSTRACTIONS (I + D principles)
# ══════════════════════════════════════════════════════

class IResearcher(ABC):        # I: segregated
    @abstractmethod
    async def research(self, topic: str) -> str: ...

class IWriter(ABC):            # I: segregated
    @abstractmethod
    async def write(self, content: str) -> str: ...

class IReviewer(ABC):          # I: segregated
    @abstractmethod
    async def review(self, draft: str) -> str: ...


# ══════════════════════════════════════════════════════
# CONCRETE AGENTS (S: each has one job, L: substitutable)
# ══════════════════════════════════════════════════════

class ResearchAgent(IResearcher):
    def __init__(self, llm=None):
        self.llm = llm or ChatOpenAI(model="gpt-4o-mini")  # D: injected

    async def research(self, topic: str) -> str:
        resp = self.llm.invoke([
            HumanMessage(content=f"Research '{topic}'. Give 5 key facts.")
        ])
        return resp.content

class WriterAgent(IWriter):
    def __init__(self, llm=None):
        self.llm = llm or ChatOpenAI(model="gpt-4o-mini")  # D: injected

    async def write(self, content: str) -> str:
        resp = self.llm.invoke([
            HumanMessage(content=f"Write a concise article from: {content}")
        ])
        return resp.content

class ReviewerAgent(IReviewer):
    def __init__(self, llm=None):
        self.llm = llm or ChatOpenAI(model="gpt-4o-mini")

    async def review(self, draft: str) -> str:
        resp = self.llm.invoke([
            HumanMessage(content=f"Review and improve this draft:\n{draft}")
        ])
        return resp.content


# ══════════════════════════════════════════════════════
# LANGGRAPH PIPELINE (O: open for extension)
# ══════════════════════════════════════════════════════

class PipelineState(TypedDict):
    messages: Annotated[Sequence[BaseMessage], operator.add]
    topic: str
    stage: str

def build_pipeline(
    researcher: IResearcher,  # D: depends on abstractions
    writer: IWriter,
    reviewer: IReviewer
):
    def research_step(state):
        import asyncio
        result = asyncio.get_event_loop().run_until_complete(
            researcher.research(state["topic"])
        )
        return {"messages": [AIMessage(content=result)], "stage": "write"}

    def write_step(state):
        research = state["messages"][-1].content
        result = asyncio.get_event_loop().run_until_complete(
            writer.write(research)
        )
        return {"messages": [AIMessage(content=result)], "stage": "review"}

    def review_step(state):
        draft = state["messages"][-1].content
        result = asyncio.get_event_loop().run_until_complete(
            reviewer.review(draft)
        )
        return {"messages": [AIMessage(content=result)], "stage": "done"}

    graph = StateGraph(PipelineState)
    graph.add_node("research", research_step)
    graph.add_node("write", write_step)
    graph.add_node("review", review_step)

    graph.set_entry_point("research")
    graph.add_edge("research", "write")
    graph.add_edge("write", "review")
    graph.add_edge("review", END)

    return graph.compile()


# ══════════════════════════════════════════════════════
# COMPOSITION ROOT — wire everything together here
# ══════════════════════════════════════════════════════
if __name__ == "__main__":
    pipeline = build_pipeline(
        researcher=ResearchAgent(),  # swap any IResearcher here
        writer=WriterAgent(),
        reviewer=ReviewerAgent(),
    )

    result = pipeline.invoke({
        "messages": [],
        "topic": "the future of agentic AI systems",
        "stage": "research"
    })

    print("═" * 60)
    print("FINAL REPORT")
    print("═" * 60)
    print(result["messages"][-1].content)

SOLID Principles Summary

quick reference summary 
S — Single Responsibility:  Each agent/tool does ONE job
                              ResearchAgent researches, WriterAgent writes

O — Open/Closed:             Add new LLM providers via subclasses
                              Never modify OpenAIAgent to add AnthropicAgent

L — Liskov Substitution:     MockSearchTool works anywhere WebSearchTool does
                              Enables testing without real API calls

I — Interface Segregation:   IResearcher, IWriter, IReviewer are separate
                              No agent forced to implement unused methods

D — Dependency Inversion:    Orchestrator depends on IResearcher, not ResearchAgent
                              Concrete agents injected at composition root
