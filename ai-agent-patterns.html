Pattern 01

ReAct — Reasoning + Acting

ReAct interleaves Thought, Action, and Observation steps in a tight loop. Each observation feeds directly into the next reasoning step, allowing the agent to correct course mid-task without external intervention.

💡
Key insight

The agent never commits to a full plan upfront. Instead it reasons one step at a time, adapting as new information arrives from tool observations.

Flow
Thought → Action → Observation ↻ loop Thought → Action → Final Answer
Example trace
react_trace.txt trace 
Thought: I need to find the population of Tokyo.
Action:  web_search("Tokyo population 2024")
Obs:     Tokyo city proper has ~13.96 million residents.

Thought: The user asked for metro area, not city proper.
Action:  web_search("Tokyo metro area population")
Obs:     Greater Tokyo Area: ~37.4 million.

Thought: I now have the correct figure.
Answer:  The Greater Tokyo metro area has ~37.4 million people.
Implementation
react_agent.py python 
def react_agent(task: str, tools: dict, max_steps: int = 10):
    history = [{ "role": "user", "content": task }]

    for step in range(max_steps):
        response = llm.complete(
            system=REACT_PROMPT,
            messages=history
        )

        # Parse Thought / Action / Answer
        parsed = parse_react(response)

        if parsed.type == "answer":
            return parsed.content   # Done

        # Execute the chosen tool
        tool_fn = tools[parsed.action]
        observation = tool_fn(**parsed.args)

        history.append({ "role": "assistant", "content": response })
        history.append({ "role": "tool", "content": observation })

    raise TimeoutError("Max steps reached")
✓ Pro

Adapts mid-task — observations can redirect reasoning before wasting more steps on a wrong path. Excellent for open-ended research tasks.

✗ Con

Can spiral into long loops. Without a step cap the agent may oscillate and burn tokens chasing irrelevant sub-questions.

Pattern 02

Plan & Execute

Separate planning from doing. A Planner LLM decomposes the goal into a concrete task list, then an Executor LLM works through each step in sequence. The plan can be inspected and edited before execution begins.

Flow
User goal → Planner LLM → Step list → Executor LLM → Result
Example
plan_execute.txt trace 
Goal: "Write a competitive analysis report on Notion vs Linear"

Plan:
  1. Search for Notion key features
  2. Search for Linear key features
  3. Retrieve current pricing for both
  4. Retrieve user sentiment / reviews
  5. Draft a side-by-side comparison table
  6. Write 300-word executive summary

Executor runs steps 1–6 in sequence,
each tool call building on the previous result.
Implementation
plan_execute.py python 
def plan_and_execute(goal: str):
    # Phase 1: Plan
    plan: list[str] = planner_llm.complete(
        prompt=f"Decompose this goal into ordered steps:\n{goal}"
    )

    results = []

    # Phase 2: Execute each step
    for i, step in enumerate(plan):
        context = format_results(results)
        result = executor_llm.complete(
            prompt=f"Previous results:\n{context}\n\nNow: {step}",
            tools=TOOL_REGISTRY
        )
        results.append({ "step": step, "result": result })
        checkpoint.save(i, results)   # persist state

    return synthesise(goal, results)
✓ Pro

Predictable and auditable — you can inspect and edit the plan before execution begins, catching bad strategies early.

✗ Con

Rigid — if an early step fails or returns surprising data, the plan doesn't automatically rewrite itself mid-execution.

Pattern 03

Reflection / Self-critique

After producing output, the agent critiques its own work and iterates until quality thresholds are met or a max revision count is hit. Works exceptionally well for writing, code generation, and analysis tasks.

Flow
Generate draft → Critique → Issues found? — yes → Revise ↻ loop Accept
Example trace
reflection_trace.txt trace 
Draft: "Python is the best language for everything."

Critique:
  - Claim is absolute and unsupported
  - No mention of tradeoffs (speed, typing, mobile dev)
  - Missing comparative context

Revision: "Python excels at data science and scripting due
to its ecosystem, but compiled languages like Rust
or Go are better suited for performance-critical
systems or low-latency services."

Critique: Improved. Claim is qualified. Accept.
Implementation
reflection.py python 
def reflection_agent(task: str, max_revisions: int = 3):
    draft = generator_llm.complete(task)

    for _ in range(max_revisions):
        critique = critic_llm.complete(
            prompt=f"Critique this output:\n{draft}",
            schema=CritiqueSchema   # structured JSON
        )

        if critique.approved:
            break

        draft = generator_llm.complete(
            prompt=f"Revise based on this critique:\n{critique.issues}",
            context=draft
        )

    return draft
⚠️
Cost consideration

Reflection runs 2–3× more LLM calls per task. Set a sensible max_revisions and use a cheaper model for the critic to keep costs manageable.

✓ Pro

Consistently raises output quality. Works well for writing, code review, and complex reasoning where first drafts are rarely optimal.

✗ Con

2–3× more expensive per task. The critic can sometimes nitpick trivialities rather than catching substantive problems.

Pattern 04

Tool-calling Agent

The LLM is equipped with a schema-defined set of tools and decides which to invoke — and in what order — at runtime. This is the backbone of most production agents today and is natively supported by all major model providers.

Flow
User message → LLM → Tool call → Tool result → LLM → Response
Defining a tool
tools.py python 
@tool
def get_weather(city: str, units: str = "celsius") -> dict:
    """Get current weather conditions for a given city."""
    return weather_api.fetch(city, units)

@tool
def web_search(query: str, max_results: int = 5) -> list[str]:
    """Search the web and return top result snippets."""
    return search_engine.query(query)[:max_results]

# The LLM receives schemas for all tools automatically
agent = Agent(tools=[get_weather, web_search, run_code, send_email])
Example exchange
tool_calling_trace.txt trace 
User: "What's the weather in Paris — should I pack an umbrella?"

LLM selects: get_weather(city="Paris", units="celsius")
Tool returns: { temp: 14, condition: "rain", humidity: 88 }

LLM response: "It's 14°C and rainy in Paris — definitely
bring an umbrella and a light jacket."
✓ Pro

Highly extensible — adding a new capability is as simple as registering a new tool definition. No architectural changes needed.

✗ Con

Tool selection can fail on ambiguous queries. The LLM may pick the wrong tool or hallucinate incorrect parameter values.

Pattern 05

RAG — Retrieval-Augmented Generation

Query a knowledge base at runtime and inject retrieved context into the prompt to ground the LLM's response. Retrieval is treated as a tool: the agent embeds the query, searches a vector store, and passes the top-k chunks as context.

Flow
User query → Embed query → Vector search → Top-k chunks → LLM + context → Grounded answer
Implementation
rag_agent.py python 
def rag_query(question: str, k: int = 3) -> str:
    # 1. Embed the user's question
    query_embedding = embedder.encode(question)

    # 2. Retrieve top-k relevant chunks
    chunks = vector_db.search(
        query=query_embedding,
        top_k=k,
        filter={ "source": "knowledge_base" }
    )

    # 3. Build grounded prompt
    context = "\n---\n".join(c.text for c in chunks)
    prompt = f"""Answer using ONLY the context below.
Context:
{context}

Question: {question}"""

    return llm.complete(prompt)
✅
When to use RAG

Use RAG whenever the agent needs to answer questions about a private knowledge base, recent documents, or domain-specific data that wasn't in the model's training set.

✓ Pro

Keeps responses grounded in real documents — dramatically reduces hallucination for domain-specific knowledge bases.

✗ Con

Retrieval quality depends entirely on chunking and embedding strategy. Bad chunking means the right information may never surface.

Pattern 06

Multi-agent — Orchestrator / Subagents

A central orchestrator delegates subtasks to specialized subagents, each with their own tools, context windows, and capabilities. Subagents can run sequentially or in parallel depending on task dependencies.

Architecture
multi_agent.py python 
class Orchestrator:
    def run(self, goal: str):
        plan = self.plan(goal)

        results = {}
        for task in plan.tasks:
            # Route to the right specialist
            agent = self.agents[task.type]
            results[task.id] = agent.run(
                task=task.description,
                context=gather_deps(task.deps, results)
            )

        return self.synthesise(goal, results)

researcher = Agent(tools=[web_search, read_url])
coder      = Agent(tools=[run_python, read_file, write_file])
critic     = Agent(tools=[])   # reasoning only

orchestrator = Orchestrator(agents={
    "research": researcher,
    "code":     coder,
    "review":   critic,
})
Example delegation
multi_agent_trace.txt trace 
User: "Build a sales dashboard for Q1 data"

Orchestrator:
  1. → Researcher: "Find KPIs for SaaS sales dashboards"
  2. → Coder:      "Parse Q1_sales.csv, compute by-region totals"
  3. → Coder:      "Generate Plotly chart for revenue by region"
  4. → Critic:     "Review chart for accuracy and clarity"
  5. → Orchestrator: Assemble final report
✓ Pro

Parallelizable and scalable — independent agents can run concurrently, and adding specialists doesn't complicate existing agents.

✗ Con

Coordination overhead is real — orchestrators can misroute tasks, and debugging failures across agents is significantly more complex.

Pattern 07

Memory Patterns

Strategies for giving agents access to information beyond the immediate context window. Different memory types serve different needs — from sub-second scratchpad state to cross-session personalisation.

Memory types

In-context
Everything lives in the prompt. Simple and zero-latency, but bounded by the token window. Best for short, self-contained tasks.
zero latency simple
External (vector DB)
Memories stored as embeddings, retrieved by semantic similarity. Enables long-term recall across sessions and users.
long-term retrieval
Episodic
A structured log of past interactions replayed selectively. Gives the agent a sense of history without full context replay cost.
history selective
Scratchpad
Working memory within a single run — intermediate results stored in a variable or temp file, not persisted between turns.
ephemeral in-run
External memory read/write
memory.py python 
# Write a memory
memory.store(
    key="user_prefs",
    value="Prefers Python over JavaScript",
    user_id=user.id
)

--- next session ---

# Retrieve relevant memories before responding
memories = memory.retrieve(
    query="Write a script to parse this CSV",
    user_id=user.id,
    top_k=3
)

# Inject into system prompt
system = f"User preferences:\n{memories}\n\nYou are a helpful assistant."
# → Agent now writes Python, not JS, unprompted
✓ Pro

External memory lets agents personalise behaviour over time without context-window limits — key for long-running assistants and agentic workflows.

✗ Con

Memory retrieval adds latency and a new failure surface. Stale or irrelevant memories can mislead the agent or introduce outdated information.

Pattern 08

Control Flow Patterns

How the agent decides what to do next — fixed sequence, LLM-driven dynamic routing, or human checkpoints. The right choice depends on how predictable your task domain is and the consequences of routing errors.

Fixed pipeline

Ingest → Classify → Enrich → Respond
✓ Pro

Deterministic and easy to test — same input always flows the same way. Simple to monitor and debug in production.

✗ Con

Cannot adapt when a step's output is unexpected, incomplete, or outside the anticipated distribution.

Dynamic routing

Input → LLM router → Path A / Path B / Path C
✓ Pro

Flexible — the agent selects the best handler for each input type at runtime, without hardcoded conditional logic.

✗ Con

Router errors cascade. A misclassified input goes down the entirely wrong path, often silently, with no obvious failure signal.

Human-in-the-loop

Draft action → ⏸ Await approval → Execute
hitl.py python 
def hitl_agent(task: str):
    proposed_action = agent.plan(task)

    # Pause and surface to human
    approval = ui.request_approval(
        action=proposed_action,
        timeout_seconds=300
    )

    if approval.approved:
        return agent.execute(proposed_action)
    else:
        return agent.revise(task, feedback=approval.reason)
✓ Pro

Catches high-stakes mistakes before they cause real-world effects — sending emails, running SQL deletions, making API calls.

✗ Con

Breaks automation; approval latency can become a severe bottleneck for high-volume or time-sensitive workflows.

Pattern 09

Reliability Patterns

Techniques that make agent outputs safe, consistent, and recoverable. These are not optional in production — they are the difference between an agent that works occasionally and one that works reliably at scale.

Structured outputs

Force schema-compliant JSON from the LLM so downstream code can parse without fragile string manipulation.

✗ Free-form output
"I think I should search
for current weather data
in Paris to answer this."
✓ Structured output
{
  "action": "get_weather",
  "args": { "city": "Paris" },
  "confidence": 0.95
}
✓ Pro

Downstream code can parse outputs reliably without fragile regex or error-prone prompt re-runs.

✗ Con

Strict schemas can cause the model to truncate reasoning or omit nuance in order to fit the required structure.

Retry / Fallback

retry.py python 
def resilient_run(task: str):
    try:
        return agent.run(task, model="claude-opus-4")
    except ToolCallError:
        log_degraded(task)
        return agent.run(task, model="claude-haiku-4-5")
    except RateLimitError:
        time.sleep(2 ** retry_count)
        return resilient_run(task)
✓ Pro

Graceful degradation — a cheaper/faster model handles failures instead of surfacing raw errors to the user.

✗ Con

Fallback outputs may be lower quality. Silent degradation can mask systemic issues in the primary model that need attention.

Checkpointing

checkpoint.py python 
for i, step in enumerate(plan):
    result = execute(step)
    checkpoint.save(step_id=i, state=result)  # persist

# On failure at step 7 → resume from step 6, not step 0
agent.resume_from(checkpoint.load(step_id=6))
✓ Pro

Long-running agents can recover from crashes or timeouts without restarting from scratch — critical for expensive multi-step workflows.

✗ Con

Adds storage overhead and requires careful state serialisation — complex to implement correctly for branching or parallel flows.

🎯
Decision guide

Choosing a Pattern

No single pattern wins across all tasks. Map your task type to the appropriate pattern before building.

Task type Recommended pattern Why
Single-step Q&A RAG or Tool-calling One retrieval or tool call is sufficient — no loop needed
Multi-step research ReAct or Plan & Execute Needs adaptive reasoning or structured decomposition
High-quality writing / code Reflection Iterative critique consistently lifts output quality
Complex multi-domain workflows Multi-agent Each domain gets a specialist; work can parallelise
High-stakes / irreversible actions Human-in-the-loop Human approval before execution prevents costly mistakes
Long-running pipelines Checkpointing + any pattern Checkpoints allow recovery without full restarts
💡
Patterns compose

Most production agents combine patterns. A multi-agent system might use ReAct internally within each subagent, with reflection on final outputs, structured outputs for inter-agent communication, and checkpointing for resilience.

Agent Frameworks

Open-source frameworks that implement these patterns so you don't have to build from scratch.

LangGraph
Graph-based stateful agents. Nodes are agent steps; edges are transitions. Built-in checkpointing and human-in-the-loop.
stateful checkpoints multi-agent
CrewAI
Role-based multi-agent framework. Define agents with roles, goals, and backstories. Crews collaborate on tasks.
multi-agent role-based crew
AutoGen
Microsoft's conversational multi-agent framework. Agents communicate via natural language messages in configurable topologies.
conversational topology microsoft
LlamaIndex
Data framework with first-class RAG and agentic pipeline support. Excellent for knowledge-intensive applications.
RAG knowledge pipelines
Pydantic AI
Type-safe agent framework. Structured outputs via Pydantic schemas. Strong validation guarantees at every step.
type-safe validation structured
Smolagents
HuggingFace's minimal agent library. Code-first approach where agents write and execute Python to complete tasks.
code-first minimal HuggingFace
